## Carry-forward standing rules (all unchanged)

These survive every sprint and override any later design instinct:

1. **Token-first.** Every value comes from `theourgia.tokens.css`. New tokens added only when the existing palette genuinely cannot express a need; new tokens land in the supplement.
2. **`--danger` is reserved for the Visibility → Public downgrade rung — nothing else.** Every "negative" state on Workshop surfaces uses care-palette tokens. There is no destructive operation in the Workshop tier that warrants red.
3. **Tradition-neutrality at the chrome level.** Theme tokens (`hellenic`, `thelemic`, base + the dark/light modes) override the palette. The Workshop is a vault of techniques from many traditions side-by-side; no chrome may privilege one tradition's iconography.
4. **`.dc.html` is the source of truth.** Match exactly per the per-component ritual: section inventory · CSS var cross-check · verbatim editorial copy · default state · variant completeness. Expand dev numeronyms in user-facing labels ("a11y" → "accessibility", "i18n" → "internationalisation", "RTL" → "right-to-left").
5. **Sealed-content discipline.** Anything backed by encryption defaults sealed and shows count only — zero plaintext leak. Use `--seal*` family for the chrome.
6. **Wellbeing copy is verbatim from designer.** Never red even for "negative" states. Opt-in only, off by default.

## New cross-cutting rules earned in H04 (must carry forward)

The H04 sprint added these — they apply to every Workshop surface as much as to Phase 06:

7. **Citation chrome on every traditional artefact.** When a surface displays text/imagery the practitioner did not author (a public-domain sigil source, a Heptameron circle preset, a `Picatrix` magic-square attribution, a Greek Magical Papyri voce), the source is surfaced via the `‡` `CitationKindBadge` from B54. The badge is small, muted, and never dominates the artefact — but never absent.
8. **Ritual-draw moment / committed-make moment.** Every Workshop surface that produces a permanent artefact (a sigil, a talisman card, a circle plan, a recorded voce) treats the "save" as a deliberate moment, not a casual button. The build side will not mid-edit a saved artefact silently — re-opening a saved item is read-only by default; the practitioner explicitly chooses "edit a new version" (creates a new row; the old version is retained).
9. **Quiet stats.** Where a surface shows accumulated work ("12 sigils made", "3 consecrated tools", "47 voce recordings"), the chrome is muted: small number + `--ink-mute` label. No celebration, no badges, no "well done."
10. **Honesty rules** (the Workshop equivalent of H04's symmetric-rune honesty): the surface must not let the practitioner declare something the artefact does not actually carry. Examples spelled out in the per-surface sections below.
11. **Client-side signing for sealed artefacts.** Any Workshop artefact the practitioner marks sealed uses the same client-side encryption UX shipped in B54 (CanonicalBytes preview · ciphertext-only-on-server promise · key never leaves device).

**Path-tracing UI (LOCKED — embedded mode toggle in the toolbar above the main viewport):**

The toolbar holds three radio-segmented states:
- **View** (default) — square + traditional planet sigil overlay rendered. Hover a cell shows that cell's number + gematria label as a small tooltip.

**Export:** SVG (primary) · PNG · PDF. The traditional sigil overlay is included by default; a toggle in the export dialog lets the practitioner export the bare grid + numerals only.

**Anticipated backend:** `voce` table (id · vault_id · name · source_text (the voce as written) · transliteration (Latin transliteration if source is non-Latin) · ipa (IPA notation) · source_citation · planetary_associations (array of planet keys) · elemental_associations (array of element keys) · linked_entity_ids (array) · created_at).

`voce_recording` table (id · voce_id · audio_attachment_id · duration_seconds · created_at · notes). Multiple recordings per voce supported (different practitioners' pronunciations · different days).

**Surface layout (LOCKED — list-and-detail composition, similar to Tool Registry):**

- **Top bar:** title + search input (searches voce names + source text) + tradition filter (segmented: All · Greek Magical Papyri · Hekate-tradition · Goetic · Thelemic · Vedic · Norse · Custom · ...) + "+ New voce" button.
- **Main viewport:** vertical list (NOT a card grid — voces are textual artefacts). Each voce row shows: voce text in source script (large, display font, ellipsis if long) · transliteration below (smaller, italic) · planetary glyphs + elemental glyphs as small chips (right-aligned) · recording count badge ("3 recordings" or "no recording yet").
- **Detail drawer (560px wide):** opens on tap.

**New-voce flow (LOCKED):**

"+ New voce" prompts:
1. **Source script** — picker (Greek · Hebrew · Latin · Coptic · Demotic · Other-with-custom-font)
2. **Voce text** — multi-line text input in the picked script (designer: respect the script's RTL-ness for Hebrew)
3. **Transliteration** — auto-suggested from the existing transliteration substrate (B53) where supported, editable
4. **IPA notation** — optional, free-text input with an IPA keyboard helper (a small picker of common IPA glyphs)
5. **Source citation** — text input — verbatim citation. Save populates the `‡` badge on the list row.
6. **Planetary associations** — multi-select picker (Sun · Moon · Mars · Mercury · Jupiter · Venus · Saturn)
7. **Elemental associations** — multi-select picker (Fire · Water · Air · Earth · Spirit)
8. **Linked entities** — multi-select picker (existing entities)

**Detail drawer sections (LOCKED):**

1. **Voce text** — the source-script text rendered large (display font sized to the script). Below: transliteration + IPA.
2. **Source citation** — the `‡` badge inline with the citation text.
3. **Associations** — planetary chips + elemental chips + linked entities list.
4. **Recordings** — list of audio recordings. Each row: a play-pause control · waveform thumbnail · duration · creation date · notes (one-line). A "+ Record new" button opens the audio capture chrome (uses the audio substrate from B34 — same chrome as scrying audio in B85). Multiple recordings supported per voce (the practitioner's pronunciation can evolve; the surface honours that lineage).
5. **Used in workings** (read-only computed) — list of workings that referenced this voce.

**Ritual-mode playback (LOCKED — referenced from elsewhere, NOT a separate surface):**

A working's entry (composed via B61 Template Designer / future Tiptap integration) can embed a voce reference. When the working is played back in a future "ritual mode" view (deferred to a later sprint), the embedded voce's preferred recording plays. The Voces Magicae Recorder surface itself only authors voces; the ritual-mode playback chrome will be designed in its own handoff when the time comes.

**Honesty rules:**

- Recordings cannot be silently overwritten. Adding a new recording appends; deleting an existing recording soft-deletes (the row remains, marked deleted, accessible from a "Show deleted recordings" admin affordance — the same discipline as entries).
- Source citation is required. The build side will not save a voce row without a citation populated. The surface enforces this by disabling Save until the citation field is non-empty.

**Library curation:**

The first batch of built-in voces ships as fixtures (PGM IV.2785–2890 Hekate hymn voces, the verbatim PD-eligible barbarous names; the Heptameron's angelic voces; etc.). Built-in voces appear in the list alongside the practitioner's own, distinguished only by a small `‡` marker. Built-in voces cannot be edited or deleted — they are read-only references. The practitioner can fork a built-in voce into a custom one (preserving the citation link).

**Export:** per-recording audio download. No bulk export from this surface (voces are not redistribution artefacts).
