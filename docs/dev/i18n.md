# i18n — developer guide

Theourgia ships with i18n from day one. Every user-facing string must flow through the translator. The patterns below cover the common cases.

## The substrate at a glance

```
core/i18n/
├── locale.py         # ContextVar: get_current_locale, bind_locale
├── negotiation.py    # parse_accept_language, negotiate_locale
├── catalog.py        # Catalog Protocol + InMemoryCatalog + BabelCatalog
├── translator.py     # Translator Protocol + _, _lazy, _n, _n_lazy
├── lazy.py           # LazyString
├── middleware.py     # LocaleMiddleware (ASGI)
└── factory.py        # build_translator_from_settings
```

## Canonical call points

```python
from theourgia.core.i18n import _, _lazy, _n, _n_lazy
```

### Inside a request handler / runtime function

```python
def my_endpoint():
    # Simple translation
    title = _("Welcome to Theourgia")

    # With interpolation — placeholders use str.format syntax
    greeting = _("Hello, {name}!", name=user.display_name)

    # Pluralization
    summary = _n(
        "You have {n} unread entry.",
        "You have {n} unread entries.",
        unread_count,
    )
```

### Module-level constants — use the lazy variants

```python
from theourgia.core.i18n import _lazy, _n_lazy

# WRONG — translation freezes to the default locale at import time:
PASSWORD_TOO_SHORT = _("Password must be at least 8 characters.")

# RIGHT — translation happens when the value is read:
PASSWORD_TOO_SHORT = _lazy("Password must be at least 8 characters.")
```

`LazyString` instances coerce to `str` anywhere a string is expected (string concatenation, `f"{...}"`, `.format()`, raising as exception messages, etc.).

### Inside Celery tasks / CLI scripts

Outside an HTTP request, no middleware has bound a locale. Code that wants a specific one calls:

```python
from theourgia.core.i18n import bind_locale, _

bind_locale("es")  # for the rest of this task
welcome = _("Welcome to Theourgia")
```

For email templates that need to render in a specific recipient's locale, the email service handles binding automatically (Phase 10).

## Marking strings for extraction

`pybabel extract` finds calls to a configured set of function names — by default `_`, `gettext`, `ngettext`. Our `babel.cfg` also recognizes the lazy variants. Just call them as functions:

```python
err = _("Token has expired.")        # extracted
err = gettext("Token has expired.")  # also extracted
```

Anything you string-build dynamically does NOT get extracted:

```python
# NOT extracted (the source is computed at runtime)
err = _("Token " + status + " expired.")  # WRONG

# RIGHT — interpolation happens AFTER translation
err = _("Token {status} expired.", status=status)
```

## Pluralization rules vary by language

English has two plural forms (1 vs not-1). Russian has 3. Arabic has 6. CLDR plural rules in Babel handle this automatically — but you must always supply the English singular AND plural forms as the source:

```python
# Source forms in English
_n("file", "files", count)

# Translators provide the right number of forms per their language's
# Plural-Forms header in the .po file. Don't try to be clever in
# Python with "{n}-singular if n == 1 else plural" — let the catalog
# decide.
```

## Testing translated strings

```python
import pytest
from theourgia.core.i18n.catalog import InMemoryCatalog
from theourgia.core.i18n.locale import bind_locale, clear_locale
from theourgia.core.i18n.translator import (
    InMemoryTranslator,
    configure_translator,
    reset_translator,
)


@pytest.fixture(autouse=True)
def _reset():
    reset_translator()
    clear_locale()
    yield
    reset_translator()
    clear_locale()


def test_my_feature_in_spanish():
    configure_translator(
        InMemoryTranslator(
            {
                "en": InMemoryCatalog("en", messages={"Hello": "Hello"}),
                "es": InMemoryCatalog("es", messages={"Hello": "Hola"}),
            },
            default_locale="en",
        )
    )
    bind_locale("es")
    assert my_feature.greet() == "Hola"
```

## Style — when to translate, when not

**Translate:**
- User-facing error messages
- UI labels, headings, page titles
- Email subjects + bodies
- Notification text
- Documentation that the user reads

**Don't translate:**
- Log messages (audited mostly by operators / developers)
- Audit-event `kind` values (those are stable identifiers consumed programmatically)
- Internal exception types
- API field names, JSON keys
- OpenAPI descriptions consumed by integrators (English is the lingua franca)

When in doubt: ask "would a user ever see this on screen?" If yes, `_()` it.

## Common gotchas

- **Don't break sentences across multiple `_()` calls.** Different languages reorder words. `_("Hello") + ", " + _("world")` is wrong; use `_("Hello, world")` or `_("{greeting}, {target}", greeting=..., target=...)`.
- **Numbers, dates, currency — use Babel's formatters**, not string concatenation: `babel.numbers.format_currency(price, "EUR", locale=get_current_locale())`. (Wrappers for these land with Phase 03 when the first feature needs them.)
- **HTML in translatable strings is OK** but discouraged. Prefer to wrap the HTML structure around `_()` calls, not inside them.

## Adding a new supported locale

1. The operator adds the locale code to `THEOURGIA_SUPPORTED_LOCALES`.
2. Translators produce `locales/<locale>/LC_MESSAGES/messages.po`.
3. Run `pybabel compile -d locales` to produce the `.mo`.
4. Restart workers.

See `docs/admin/i18n.md` for the operator-side commands.
