# Internationalization — operator runbook

Theourgia supports per-instance locale configuration. Operators pick which languages the instance serves; users get content in their preferred locale via the standard `Accept-Language` HTTP header.

## Configuration

```bash
THEOURGIA_DEFAULT_LOCALE=en
THEOURGIA_SUPPORTED_LOCALES=en,es,fr,pt-BR,ja
THEOURGIA_LOCALES_PATH=backend/locales
```

- **`DEFAULT_LOCALE`** — fallback when no `Accept-Language` candidate matches a supported locale, and the locale a user sees if their browser sends no preference.
- **`SUPPORTED_LOCALES`** — comma-separated list. The instance only serves these.
- **`LOCALES_PATH`** — filesystem path to the Babel `.mo` catalogs. Default works for the standard repo layout.

## Adding a translation

1. Extract the source messages from the codebase:

   ```bash
   cd backend
   pybabel extract -F babel.cfg \
     -k _ -k _lazy -k _n -k _n_lazy \
     -k gettext -k gettext_lazy -k ngettext -k ngettext_lazy \
     -o locales/messages.pot theourgia/
   ```

2. Initialize the new locale (only once per locale):

   ```bash
   pybabel init -i locales/messages.pot -d locales -l es
   ```

3. Translate the `.po` file (Weblate, Crowdin, by hand, etc.).

4. Compile the `.po` into a runtime `.mo`:

   ```bash
   pybabel compile -d locales
   ```

5. Restart the backend so the new catalog is picked up. (Translator construction reads the directory at process start; existing workers don't hot-reload catalogs.)

## How locale negotiation works

For each request:

1. If the URL has `?locale=xx` and `xx` is in `SUPPORTED_LOCALES`, use it. (Useful for sharing a localized link out-of-band.)
2. Else, parse the `Accept-Language` header and pick the highest-quality supported match. Language-only prefix matches count (e.g., a user requesting `en-US` gets served `en` if the instance only supports `en`).
3. Else, fall back to `DEFAULT_LOCALE`.

Users who are logged in eventually see content in their **stored preferred locale** if they've set one (Phase 03 adds the `User.preferred_locale` column). The stored preference overrides `Accept-Language` for authenticated requests.

## What happens when a translation is missing

A missing `.mo` file, a missing message inside a `.mo`, or a missing plural form all degrade to the source string (English by default). The system never emits an empty string or an error to the user — at worst, a non-English-speaker sees an English message. This is intentional: missing translations are an operational issue to fix, not a hard failure.

## Diagnostics

The Prometheus metrics surface (Phase 9) includes:

- `theourgia_i18n_translations_missing_total{locale,message}` — counter of source-string passthroughs. Use this to find what your translators still need to cover.

(That metric lands when the first feature actually starts emitting translatable strings; for now, the substrate is in place but no metric counter increments.)

## Supported locales today

In the default install, only `en` (the source locale) is fully populated. The substrate accepts any BCP 47 tag — operators can target whatever languages their community needs. The Theourgia project itself accepts translation contributions via the GitHub repo.
